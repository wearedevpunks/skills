Framework skills in this repo are meant to vendor into many projects, so path guidance must be setup-agnostic. Hard-coded workspace names like `packages/api`, `packages/auth`, or repo-local "this repo" phrasing mislead downstream users into treating one host project's structure as normative. Prefer placeholders like `<backend-root>`, "backend workspace/app", and "workspace-local instructions" unless the exact framework convention truly requires a concrete path.
